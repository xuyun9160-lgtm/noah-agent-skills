# Order Status Mapping

## 1. Purpose
交易 OpenAPI 中的订单状态包含两层：
- OpenAPI 统一后的状态，例如：`SUBMITTED`、`PARTIAL_FILLED`、`FILLED`、`CANCELED`、`FAILED`
- 底层 sec-trade 原始状态码，例如：`0`、`1`、`2`、`3`、`4`、`7`、`8`、`9`、`A`、`B`、`H`、`W` 等

本文件用于把底层原始状态码整理成更适合用户理解的解释。

## 2. OpenAPI Unified Status
当前文档中出现的统一状态包括：
- `SUBMITTED`：已提交 / 处理中
- `PARTIAL_FILLED`：部分成交
- `FILLED`：全部成交
- `CANCELED`：已撤销
- `FAILED`：失败 / 废单

## 3. Raw Status Mapping

| Raw Status | Meaning | Suggested User-facing Interpretation |
|---|---|---|
| `0` | 未成 | 订单已报出，当前尚未成交 |
| `1` | 待报 | 订单已提交，等待报送市场 |
| `2` | 已报 | 订单已报送市场，等待成交 |
| `3` | 已报待撤 | 已提交撤单请求，等待撤单完成 |
| `4` | 部成待撤 | 订单部分成交，剩余部分正在撤销 |
| `5` | 部撤 | 订单部分成交，其余部分已撤 |
| `6` | 已撤 | 订单已完全撤销 |
| `7` | 部成 | 订单已部分成交 |
| `8` | 已成 | 订单已全部成交 |
| `9` | 废单 | 订单失败或被交易系统拒绝 |
| `A` | 已报待改 | 已提交改单请求，等待修改完成 |
| `B` | 待复核【人手买方待确认】 | 订单待人工复核 / 确认 |
| `C` | 重新检查中 | 订单正在重新检查处理中 |
| `D` | 待触发 | 条件未满足，订单尚未触发 |
| `E` | 部成待改 | 订单部分成交，剩余部分正在修改 |
| `F` | 撤单(过期) | 撤单请求因时效原因未完成 |
| `G` | 部撤(过期) | 部分撤销，但剩余流程已过期 |
| `H` | 待复核 | 订单待复核 |
| `J` | 复核未通过 | 订单复核未通过 |
| `W` | 待确认 | 订单待用户或系统确认 |
| `N` | 修改待复核 | 改单请求待复核 |
| `O` | 修改待复核【部成】 | 部分成交订单的改单请求待复核 |
| `P` | 修改确认 | 改单请求已确认 |

## 4. Product-level Grouping
为了减少最终用户面对大量原始状态码时的认知负担，后续用户态展示可按以下分组理解：

### 待处理 / 处理中
- `1` 待报
- `2` 已报
- `3` 已报待撤
- `A` 已报待改
- `B` 待复核
- `C` 重新检查中
- `H` 待复核
- `W` 待确认
- `N` 修改待复核
- `O` 修改待复核【部成】
- `P` 修改确认

### 部分成交
- `4` 部成待撤
- `5` 部撤
- `7` 部成
- `E` 部成待改
- `G` 部撤(过期)

### 已完成
- `6` 已撤
- `8` 已成

### 失败 / 异常
- `9` 废单
- `F` 撤单(过期)
- `J` 复核未通过

### 条件单 / 特殊状态
- `D` 待触发

## 5. Output Suggestion
后续在用户态输出时，建议：
- 优先展示 OpenAPI 统一状态
- 若需要补充细节，再追加原始状态码解释
- 避免直接把原始状态码裸暴露给最终用户而没有说明

## 6. Notes
- 原始状态码来自交易 OpenAPI 文档，后续若底层 sec-trade 状态定义变动，应同步更新本文件。
- 推送数据中的交易日志状态码与订单状态码不是完全同一套，后续如需要可再单独补 `push-status-mapping.md`。
